# Model-Builder — Этап 4

**Вход:** драйверы рва → **Выход:** причинная петля компаундинга + проверка «разрыв со временем растёт?», с kill-condition на каждый драйвер.

## Что делает
Из драйверов команды собирает причинную петлю: драйвер → накопление → разрыв с конкурентом растёт → драйвер усиливается. Затем проверяет, действительно ли петля компаундит (разрыв растёт, а не схлопывается), и требует на каждый драйвер три обязательных поля. Структуру даёт ИИ, истинность модели подтверждает только устойчивость во времени — вердикт не в зале.

## Вход
- Выбранная на Этапе 3 гипотеза + её тип рва (REFERENCE.md §Типология).
- Список драйверов — что, по версии команды, накапливается со временем.
- Поля карты: что накапливается (5), эксклюзивный доступ (7), издержки ухода (6).
- Опционально — что red-team-copy НЕ смог скопировать за квартал.

## Промпт-цепочка
1. **Старт с презумпции истощения** — по умолчанию «время против нас», преимущество mean-reverting (REFERENCE.md §Сквозная рамка). Модель обязана это **перебить** конкретным механизмом, а не предположить отрыв. Зафиксируй презумпцию первой строкой.
2. **Разбери каждый драйвер на три обязательных поля (§8.3):**
   - **Непубличный накапливаемый механизм** — что именно копится и почему этого нет в открытом доступе / не реплицируется ИИ за квартал.
   - **Внешний проверяющий** — кто/что вне команды подтвердит накопление (клиент, рынок, метрика во внешнем канале), не «мы сами видим».
   - **Kill-condition** — при каком наблюдении драйвер признаётся несработавшим (адресно: «этот драйвер этим способом в этом окне не накапливается»).
3. **Отклони драйвер без kill-condition** как «салфеточный флайвил» — красивую неопровержимую петлю. Без kill-condition драйвер в модель не входит.
4. **Собери петлю** — стрелками: драйвер_i → накопление → разрыв растёт → усиление драйвера_i (или другого). Покажи, где петля замыкается и компаундит.
5. **Проверка компаундинга** — на каждом витке разрыв с конкурентом РАСТЁТ или плато/схлопывается? Если не растёт — модель презумпцию не перебила, честно скажи.
6. **Осада «конкурент с бюджетом»** — где петля рвётся под атакой ресурсного конкурента (свяжи с red-team-copy). Самое слабое звено — пометь.

## Формат выхода
```
ИИ-черновик (model-builder). Истинность модели — только устойчивость во времени; вердикт не в зале.

Презумпция: время против нас (mean-reversion). Чем модель перебивает: <механизм>.

ДРАЙВЕРЫ
| Драйвер | Непубличный накапл. механизм | Внешний проверяющий | Kill-condition                  | В модель? |
|---------|------------------------------|---------------------|---------------------------------|-----------|
| D1      | размеченный руками корпус     | NDR клиентов        | NDR≤100% за 2 кв → не копится    | да        |
| D2      | «мы быстрее итерируем»         | —                   | —                               | НЕТ: салфеточный флайвил |

ПЕТЛЯ
D1 → больше данных на юзера → точнее продукт → выше retention → больше данных (разрыв РАСТЁТ ✓)

Проверка компаундинга: разрыв растёт на витке D1; на D2 — не оценить (отклонён).
Слабое звено под осадой «конкурент с бюджетом»: <…>.
Открытые вопросы / уверенность: <…>.
```

## Gate
- **На каждый драйвер три поля обязательны:** непубличный накапливаемый механизм + внешний проверяющий + kill-condition.
- **Драйвер без kill-condition отклоняй** как «салфеточный флайвил» — он в модель не входит.
- **Стартуй с презумпции «время против нас»** — модель должна перебить её механизмом, а не предположить отрыв. Если не перебивает — скажи прямо.
- Опровержение адресное: kill-condition убивает «этот драйвер этим способом в этом окне», не «ров вообще».
- Свой вклад помечай «ИИ-черновик/аргумент». Вердикт НЕ выносишь — его выносит команда/фасилитатор.
- Faithfulness · traceability · uncertainty: не достраивай отсутствующий механизм красивой петлёй; нет механизма — пиши «механизм не назван».
