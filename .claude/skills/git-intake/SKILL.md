# git-intake — карта продукта из репозитория — вход Фазы A

**Альтернатива ручному опроснику.** Фаундер даёт ссылку на git-репозиторий своего продукта → ты читаешь репо и заполняешь `canvases/00-product-map.md` сам, как ЧЕРНОВИК. Потом показываешь фаундеру и дозаполняешь то, чего в коде нет.

## Что искать: код И материалы

В репозитории два пласта источников — читай оба:

- **Код** — что продукт делает и как устроен (фичи, стек, модель данных).
- **Материалы** — рыночное и продуктовое знание, которого в коде нет: `docs/`, `research/`, README-разделы, питч-деки, бизнес-планы, заметки с интервью, GTM/маркетинг, анализ конкурентов, ADR, CHANGELOG, метрики-доки, борды/роадмапы.

**Материалы часто закрывают поля, которые код закрыть не может.** Поэтому правило не «код vs выдумка», а: **есть в репо (код или док) → бери с источником и пометкой «подтвердить»; нет нигде → `❓ нужен ответ фаундера`, не выдумывай.**

### Откуда что берётся

| Поле карты | Где искать | Если нет в репо |
|---|---|---|
| #1 Однострочник | README / package.json / лендинг | ❓ |
| #2 ЛПР (платит/решает) | docs, интервью, GTM, ICP-доки | ❓ |
| #3 Статус-кво | README «вместо чего», research | ❓ |
| #4 **Что копируется за квартал** | **код** — фичи/стек/архитектура = слой C | (почти всегда есть) |
| #5 Что накапливается | модель данных/схемы + продуктовые доки | кандидат, проверить |
| #6 Издержки ухода | docs, интервью, описание интеграций | ❓ |
| #7 Эксклюзивный доступ | docs о партнёрствах/данных/лицензиях | ❓ |
| #8 Каналы + CAC | GTM/маркетинг-доки, аналитика | ❓ |
| #9 Метрики | метрики-доки, дашборд-экспорты, research | ❓ — **числа без источника не подставляй** |
| #10 Что инкумбент не повторит | питч, конкурентный анализ | гипотеза, проверить |
| #11 Наивная гипотеза рва | маркетинг-копи, питч | ❓ |
| Стадия | CHANGELOG, доки о клиентах/выручке | ❓ — уточни (есть ли платящие) |

**Жёстко:** число (метрика, CAC) ставь только если в репо есть документ-источник — со ссылкой на него. Нет дока → `❓`, иначе построишь фейковый ров.

## Процедура

1. **Получи доступ.** Если репо доступен — клонируй (read-only) или читай через web. Приватный/недоступный → попроси сделать публичным временно или вставить README + структуру.
2. **Сними карту репозитория.** Сначала структура (`git ls-files` / дерево папок) — найди оба пласта: где код, а где **материалы** (`docs/`, `research/`, `*.md` вне кода, питч/`*.pdf`, заметки, `notes/`, `gtm/`, `competitors*`). Материалы по рынку/продукту читай в первую очередь — в них поля #2/#8/#9, которых в коде нет.
3. **Прочитай в порядке сигнала:** README → материалы по рынку/продукту (docs, research, питч, интервью, конкуренты, метрики-доки) → лендинг/маркетинг-копи → package.json/pyproject → модели данных/схемы БД (migrations, models, prisma, sql) → основные фичи в коде. Не читай весь код — ищи: что делает продукт, какие фичи (слой C), что накапливается, и что материалы говорят про рынок.
4. **Заполни карту** `canvases/00-product-map.md`:
   - извлекаемые поля — черновик + **источник** в скобках: `(README)`, `(prisma/schema)`, `(landing)`;
   - неизвлекаемые — `❓ нужен ответ фаундера`;
   - помечай весь свой ввод как **ИИ-черновик из репо**.
5. **Покажи фаундеру заполненную карту** и спроси прямо:
   - подтверди/поправь извлечённое (особенно #4 — полный ли список копируемого, и #5 — что реально накапливается);
   - дай ответы на ❓-поля (метрики, ЛПР, каналы, эксклюзив, стадия) — без них тест неполноценен.
6. **Gate перед переходом к Фазе B:** ☐ #4 подтверждён фаундером ☐ #9 метрики даны числами или явное «нет данных» ☐ стадия определена ☐ ❓-полей не осталось (или помечены «неизвестно» осознанно).

## Репозиторий — источник на весь диалог (не только вход)

git-intake снимает **первый** срез. Дальше, на всех 6 этапах, репо остаётся живым источником — возвращайся к нему, не опирайся только на снимок входа:

- На этапе **аналогов/гипотез** — перечитай research/конкурентов в репо, прежде чем выдумывать.
- На **фильтре фейков** — проверь по коду, действительно ли «фича» это слой C (загляни, насколько она тонкая/толстая в реализации).
- На **модели** — драйверы накопления сверяй с реальной моделью данных (что схема правда копит).
- На **метриках** — ищи в репо доки/экспорты с числами, прежде чем просить у фаундера.
- Когда фаундер в диалоге упоминает факт — проверь, есть ли он в репо (commit, doc, схема), и попроси добавить, если нет.

Помечай каждое новое обращение к репо как источник (`файл:строка`). Репо обновился по ходу — перечитай изменённое.

## Формат выхода

```
ИИ-черновик карты продукта из репозитория <url> (проверяет фаундер)

1. Однострочник: <…> (README)
3. Статус-кво: <…> (README) | ❓ нужен ответ
4. Что копируется за квартал: <фичи/стек из кода, честно> (src, package.json)
5. Что накапливается: <кандидаты из схемы данных> — кандидат, проверить (prisma/schema.prisma)
6. Издержки ухода: ❓ нужен ответ фаундера
7. Эксклюзивный доступ: ❓ нужен ответ фаундера
8. Каналы + CAC: ❓ нужен ответ фаундера
9. Метрики: ❓ нужен ответ фаундера — ЧИСЛА из кода не извлекаются
10. Что инкумбент не повторит: <гипотеза из позиционирования> — проверить
11. Наивная гипотеза рва: <из маркетинг-копи> (landing/README)

Стадия: ❓ есть ли платящие клиенты? (из кода не видно)

Дальше: подтвердите #4 и #5, ответьте на ❓-поля — и пойдём по 6 этапам.
```

> Помни (CLAUDE.md): faithfulness — чего нет в репо, помечай «неизвестно», не достраивай; separability — весь твой ввод помечен «ИИ-черновик из репо»; privacy — не выводи наружу секреты/.env, читай только то, что нужно для карты.
